# site-wraper
pet-проект с оболочками сайтов

## Формулировка
Разработчику часто приходится работать с веб-интерфейсами, которые меняются быстрее, чем их функциональность. В результате теряется мышечная память, снижается эффективность и возникает раздражение. Задача — создать инструмент, который позволит «заморозить» привычный вид сайта, даже если исходный код интерфейса изменился. Инструмент должен работать как локальный прокси, подменяя HTML/CSS/JS на лету, и при этом быть простым в развёртывании, тестировании и мониторинге.

## Цель
Создать прототип кастомной оболочки для веб-сайтов на основе локального прокси-сервера, позволяющий пользователю фиксировать и восстанавливать привычный внешний вид интерфейса при изменении исходного сайта. Проект должен охватывать полный DevOps-цикл: контейнеризацию, CI/CD и мониторинг.

## Отрабатываемый материал
DevOps-практики (контейнеризация, CI/CD, мониторинг), компьютерные сети (прокси, MITM), программирование (Python/Go), системное администрирование (Docker, автоматизация), безопасность (TLS, CSP).

# Задание
Разработать систему, состоящую из:
  1) Прокси-сервера с возможностью модификации трафика (на базе mitmproxy на Python или кастомного прокси на Go).
  2) Механизма адаптеров — набора правил (CSS, JS, модификации DOM) для целевых сайтов.
  3) Системы контейнеризации и оркестрации (Docker, Docker Compose).
  4) CI/CD-пайплайна (GitHub Actions / GitLab CI) для автоматической сборки, тестирования и развёртывания.
  5) Базового мониторинга (Prometheus + Grafana) с метриками работы адаптеров.

## Функциональные требования

### 1. Прокси-сервер
  - Прокси должен принимать HTTP/HTTPS-трафик от браузера, перенаправлять его на целевой сайт и модифицировать ответы согласно активным адаптерам.
  - Прокси должен уметь расшифровывать HTTPS-трафик (MITM) с помощью самоподписанного сертификата, который пользователь добавляет в доверенные.
  - Прокси должен поддерживать динамическую загрузку адаптеров без перезапуска (например, через горячую перезагрузку конфигов).

### 2. Адаптеры
  - Адаптер — это набор правил для одного домена (например, `github.com`).
  - Адаптер должен содержать:
    - список обязательных CSS-селекторов, проверяющих актуальность сайта;
    - кастомные CSS-стили для восстановления внешнего вида;
    - JavaScript-код для модификации DOM (перемещение/скрытие блоков);
    - (опционально) модификацию HTTP-заголовков (CSP, CORS).
  - Если хотя бы один обязательный селектор не найден на странице, адаптер не применяется, а в лог выводится предупреждение (и, опционально, показывается уведомление пользователю).
  
### 3. Контейнеризация
  - Весь сервис (прокси + адаптеры + мониторинг) должен быть упакован в Docker-образы и управляться через `docker-compose up`.
  - Адаптеры должны монтироваться как volume, чтобы их можно было обновлять без пересборки образа.

### 4. CI/CD
  - При пуше в репозиторий должен запускаться пайплайн, который:
    - проверяет синтаксис и запускает юнит-тесты адаптеров (например, проверку, что обязательные селекторы действительно существуют на целевой странице);
    - собирает Docker-образ;
    - (опционально) деплоит его на тестовый стенд или локально перезапускает контейнер.

### 5. Мониторинг
  - Прокси-сервер должен экспортировать метрики в формате Prometheus:
    - `adapter_hits_total{site}` — количество успешных применений адаптера;
    - `adapter_failures_total{site, reason}` — количество ошибок (например, пропавшие селекторы);
    - `request_duration_seconds` — гистограмма времени обработки запросов.
  - Grafana должна предоставлять дашборд для визуализации этих метрик.

## Нефункциональные требования
- Проект должен быть реализован с использованием принципов ООП и чистой архитектуры (разделение на модули: прокси-ядро, загрузчик адаптеров, экспортёр метрик).
- Конфигурация адаптеров должна храниться в человеко-читаемом формате (YAML/JSON) и быть версионируемой в Git.
- Проект должен быть документирован: инструкция по установке, настройке и добавлению нового адаптера.
- Логирование должно быть структурированным (JSON) для последующего анализа.
- Проект не должен требовать прав администратора для установки (кроме добавления сертификата в доверенные).

## Тестовые сценарии
1. **Базовый адаптер для статического сайта**  
   Выберите любой сайт (например, `example.com`). Создайте адаптер, который меняет цвет фона и скрывает определённый элемент. Проверьте, что при заходе через прокси изменения применяются.
2. **Адаптер для динамического сайта (SPA)**  
   Возьмите сайт на React/Vue (например, старый интерфейс GitHub). Создайте адаптер, который перемещает блок навигации. Проверьте, что изменения сохраняются после навигации внутри SPA (MutationObserver).
3. **Падение адаптера**  
   Измените селектор в адаптере на несуществующий. Убедитесь, что прокси выдаёт предупреждение в лог и не применяет модификации, а сайт остаётся нетронутым.
4. **Метрики**  
   После нескольких запросов проверьте, что в Prometheus появились метрики `adapter_hits_total` и `adapter_failures_total` с правильными лейблами.
5. **CI/CD**  
   Внесите изменение в адаптер, запушьте в репозиторий и убедитесь, что пайплайн отработал успешно (тесты прошли, образ собран).
6. **Docker-запуск**  
   Выполните `docker-compose up` на чистой машине, настройте браузер на прокси и убедитесь, что всё работает без дополнительных ручных действий

# Этапы
## 🟢 Условия «Минимум» 

> Проект считается выполненным на минимальном уровне, если реализовано всё перечисленное ниже.

1. **Прокси-сервер**:
   - Работает в режиме HTTP/HTTPS прокси с ручной настройкой браузера.
   - Умеет расшифровывать HTTPS через MITM (самоподписанный сертификат).
   - Модифицирует HTML-ответы согласно правилам адаптера.

2. **Адаптер для одного сайта**:
   - Реализован адаптер для одного конкретного сайта (например, `github.com` или `habr.com`).
   - Адаптер содержит минимум 3 обязательных CSS-селектора для проверки актуальности.
   - Применяет кастомные CSS-стили (меняет цвета, шрифты или позиционирование).

3. **Fallback-поведение**:
   - Если хотя бы один обязательный селектор не найден — адаптер не применяется.
   - В консоль / лог выводится сообщение: `[WARN] Adapter for {site} is outdated`.

4. **Запуск без контейнеризации**:
   - Прокси можно запустить локально командой (например, `python proxy.py`).
   - Проект содержит минимальную инструкцию по запуску.

5. **Код**:
   - Код написан на выбранном языке (Python/Go) с соблюдением базового ООП.
   - Есть хотя бы один класс (например, `ProxyHandler` или `SiteAdapter`).

## 🔵 Условия «Максимум» (полноценный DevOps-проект)

> Всё из «Минимума» **плюс** пункты ниже.

1. **Поддержка нескольких сайтов**:
   - Реализована система динамической загрузки адаптеров из отдельных файлов (YAML/JSON).
   - Адаптеры для минимум 3 разных сайтов (разной структуры: статический, SPA, с авторизацией).

2. **Контейнеризация**:
   - Весь сервис упакован в Docker-образ.
   - Написан `docker-compose.yml`, поднимающий прокси + Prometheus + Grafana.
   - Адаптеры монтируются как volume для горячего обновления.

3. **CI/CD пайплайн**:
   - Настроен GitHub Actions / GitLab CI.
   - Пайплайн запускает тесты адаптеров (проверка существования селекторов).
   - Собирает Docker-образ и пушит его в registry (Docker Hub / GitHub Container Registry).

4. **Мониторинг**:
   - Прокси экспортирует метрики в Prometheus (минимум 3 метрики: хиты, ошибки, latency).
   - Настроен дашборд в Grafana с визуализацией этих метрик.
   - Настроен алерт (например, в Telegram) при частых падениях адаптера.

5. **Тестирование**:
   - Написаны юнит-тесты для загрузчика адаптеров.
   - Написаны интеграционные тесты, проверяющие работу прокси на реальном сайте (можно мокнуть HTTP-ответы).

6. **Безопасность**:
   - Прокси корректно обрабатывает CSP-заголовки (не ломает сайт).
   - Генерация CA-сертификата автоматизирована (скрипт или make-команда).

7. **Документация**:
   - Подробный `README.md` с инструкцией по установке, настройке браузера, добавлению нового адаптера.
   - Пример адаптера в формате YAML с комментариями.

8. **Дополнительные улучшения (на выбор)**:
   - Веб-интерфейс для просмотра логов и статуса адаптеров (Flask/FastAPI).
   - Возможность «обучения»: пользователь кликает на элементы старого и нового интерфейса — система запоминает соответствие.
   - Поддержка WebSocket (для сайтов с real-time обновлениями).
